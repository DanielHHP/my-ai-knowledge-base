# AI 知识库 · 项目愿景 v1.0

## 要做什么
- **数据源**: 每日抓取 GitHub Trending 总榜 Top 50（每日榜单）
- **AI项目筛选**: 使用智能匹配（NLP）判断项目是否与AI相关，检查字段包括 topics、name、description、readme 等
- **分析维度**: 使用 Agent 分析每个项目，提取以下字段：
  - `id` — 唯一标识（灵活规则，可选用 UUID 或组合标识）
  - `repo_name` — 仓库名
  - `repo_url` — 仓库链接
  - `trending_date` — 上榜日期（格式 YYYY-MM-DD，UTC时区）
  - `summary` — 项目简介（无严格字数限制）
  - `core_features` — 核心功能（数组，允许空数组）
  - `tech_stack` — 技术栈（数组，允许空数组）
  - `innovation` — 创新点（可选字段，允许 null）
  - `use_cases` — 适用场景（数组，允许空数组）
- **输出格式**: 结构化 JSON 数组，存储为纯 JSON 文件

## 不做什么
- 不分析 GitHub Trending 以外的数据源（如 Hacker News、论文、博客）
- 不处理历史数据或超过 Top 50 的排名
- 不对项目进行深度代码分析、安全审计或性能评估
- 不提供 UI 界面、API 服务或实时通知
- 不进行跨语言翻译（多语言内容保留原样）
- 不包含人工审核或质量验证流程

## 边界 & 验收
### 数据存储边界
- 输出存储路径: `knowledge/articles/YYYY-MM-DD.md`（纯 JSON 数组格式）
- 数据保留: 最近 30 天数据，旧数据可自动清理或归档
- 异常存储: 失败时输出异常信息到 `knowledge/incidents/YYYY-MM-DD.md`

### 执行时间边界
- 抓取频率: 每天 UTC 00:00:00 自动执行一次
- 执行耗时: 单日任务总耗时 < 10 分钟
- 失败重试: 仅对连接超时类错误重试，最多 3 次；其他错误直接记录异常

### Agent 分析边界
- AI 服务: 使用 DeepSeek 远端模型服务
- 成本控制: Token 成本控制在 $0.5/天以内
- 分析深度: 仅进行基础维度分析，不进行深度评估
- 字段规则: 灵活规则，字段可选，允许空数组或 null 值

### 验收标准（宽松）
- 成功标准: 只要每日任务执行完成（无论抓取或分析结果数量）即算成功
- 监控方式: 仅监控执行状态（成功/失败），不验证数据质量
- 异常处理: 任务失败时需生成完整的异常信息文件

## 怎么验证
### 监控机制
- **执行状态监控**: 每日检查任务是否成功执行
  - 成功: `knowledge/articles/YYYY-MM-DD.md` 文件存在且非空
  - 失败: `knowledge/incidents/YYYY-MM-DD.md` 文件存在，包含错误详情
- **异常告警**: 通过异常文件的存在触发告警（可集成外部监控工具）
- **日志记录**: 执行日志记录关键节点（开始抓取、分析完成、写入文件等）

### 验证方法
1. **自动化检查**: 每日任务完成后，验证输出文件格式是否为有效 JSON
2. **手动抽查**: 必要时可手动抽查分析结果的合理性和准确性
3. **成本监控**: 监控每日 token 消耗，确保不超过 $0.5 预算

### 成功指标
- **主要指标**: 任务执行成功率（月度成功率 ≥ 95%）
- **辅助指标**: 平均执行时间、异常发生频率、成本控制达标率

## 其他考虑
- **部署方式**: 待技术设计阶段确定（本地脚本、云函数、容器化等）
- **运行环境**: 需支持 Python、网络访问（GitHub API、DeepSeek API）、配置文件管理
- **手动触发**: 除定时任务外，可考虑支持手动触发执行
- **扩展性**: 当前设计为最小可行产品，未来可根据需求扩展数据源、分析维度或输出格式
